# Ітерація «Повний контроль» — дизайн

Дата: 2026-07-14 · Статус: на ревʼю у власника
Контекст: Kapot Tracker переходить у режим hosted SaaS (власник хостить сервіс для людей). Ця ітерація закриває головний UX-борг — записи й інтервали не можна переглянути детально та відредагувати — і кладе перший фундамент продакшн-надійності: бекапи та Alembic.

## 1. Мета

Користувач може переглянути, відредагувати й доповнити фотографіями **будь-що** у своїх даних без видалення і повторного створення. Дані захищені щоденним бекапом.

## 2. Скоуп

Затверджено власником (2026-07-14): детальна сторінка запису + редагування там; редагування інтервалів; фото до записів; пошук по журналу; швидкий пробіг з дашборда; дублювання запису. Додано за рішенням про hosted SaaS: Alembic і щоденний бекап у Telegram.

### 2.1 Бекенд

**Фото записів.**
- Таблиця `log_photos`: `id`, `log_entry_id` (FK → log_entries, CASCADE), `filename`, `content_type`, `size`, `created_at`. Кілька фото на запис.
- Файли: `backend/uploads/<user_id>/<uuid>.<ext>` (у `.gitignore`; named volume у compose).
- `POST /api/logs/{log_id}/photos` — multipart `file`, тільки `image/*`, ліміт 10 МБ → 201 `{id, filename, content_type, size, created_at}`. Помилки: 415/413, 404 для чужого запису.
- `GET /api/photos/{photo_id}` — стрімить файл власнику; 404 інакше.
- `DELETE /api/photos/{photo_id}` — 204, видаляє і рядок, і файл з диска.
- `LogEntry`-відповіді отримують `photos: PhotoMeta[]`.
- OCR-бонус: після успішного створення заправки зі сканованим чеком фронтенд завантажує той самий файл як фото запису.

**Пошук.** `GET /api/cars/{car_id}/logs?q=` — регістронезалежний LIKE по `notes`, `maintenance.items` (JSON як текст), `repair.category`, `repair.part_name`, `refuel.gas_station`. Комбінується з фільтром `type`. Працює на SQLite і PostgreSQL (cast JSON → String).

**Alembic.**
- `alembic init` + базова міграція, що фіксує поточну схему; друга міграція додає `log_photos`.
- Старт бекенда: `alembic upgrade head` замість «create_all + ensure_schema» (ensure_schema лишається як фолбек для старих dev-баз, викликається після upgrade).
- README: розділ про міграції; compose: upgrade в entrypoint.

**Бекап.**
- Модуль `app/backup.py`: для SQLite — копія файлу `backups/kapot_tracker-YYYYMMDD-HHMM.db`; для PostgreSQL — `pg_dump`. Ротація: останні 14.
- Доставка: якщо бот сконфігурований і користувач привʼязаний — бот надсилає архів документом у чат (sendDocument).
- Запуск: щоденна задача в циклі бот-процесу + CLI `python -m app.backup` для cron у проді.

### 2.2 Фронтенд

**Маршрут `/logbook/:id` — LogDetail.**
- Режим перегляду: всі поля запису за типом (повний список робіт ТО з розбивкою деталі/робота; категорія, деталь і гарантія ремонту; літри/ціна/повний бак/АЗС заправки), повні нотатки, галерея фото (тап → повноекранна модалка), метадані (створено, id).
- Дії: Редагувати · Повторити · Видалити (ConfirmDialog).
- Режим редагування: та сама сторінка перемикається на форму; тип запису зафіксований. Збереження → PATCH → повернення в перегляд з тостом.
- Рядок у журналі стає клікабельним (навігація на деталі); кнопка видалення лишається.

**Спільна форма.** З `AddEntry.jsx` виноситься `EntryForm` (усі type-специфічні поля, валідація, авто-математика заправки). `AddEntry` = створення (`?type=`, `?from=<id>` для дублювання: передзаповнення з запису, дата = сьогодні, пробіг = поточний, фото не копіюються). `LogDetail` = редагування через ту саму форму.

**Інтервали.** Олівець на рядку інтервалу в Гаражі → `IntervalForm` з `initial` → PATCH.

**Швидкий пробіг.** На дашборді біля одометра — олівець → інлайн-інпут (numeric), Enter/кнопка → PATCH авто. Значення менше поточного — попередження в ConfirmDialog («точно зменшити?»), бо це легальний кейс (виправлення одруку), але частіше помилка.

**Пошук.** Поле над фільтрами журналу, debounce 300 мс, поєднується з чіпсами типів; підпис «Знайдено N з M».

**Примітиви.** `Modal`, `ConfirmDialog` у `components/UI/`; усі `window.confirm` замінюються.

## 3. Тести

- Бекенд: фото (upload/чужий 404/415/413/каскадне видалення файлу з диска), пошук (нотатки, items, регістр, поєднання з type), PATCH-редагування кожного типу запису, Alembic upgrade на порожній базі і на базі зі старою схемою, бекап (файл створюється, ротація ріже старі).
- Фронтенд: vitest на `EntryForm` prefill (редагування і дублювання), на хелпер пошуку/дебаунса.

## 4. Поза скоупом

CSV-експорт/імпорт, скидання пароля, rate limiting, офлайн-черга, категорії витрат, чіпси АЗС — наступні ітерації («Дані в безпеці» частково стартувала тут бекапом і Alembic).

## 5. Ризики

- Паралельна Claude-сесія редагує `app/services/ocr.py` і `tests/test_ocr.py`. Імплементацію не починати, поки її роботу не закомічено або зони не розділено; агентам заборонено чіпати ці два файли.
- Alembic поверх ensure_schema: базову міграцію генеруємо з моделей і проганяємо на копії реальної dev-бази перед мержем.
