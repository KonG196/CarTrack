# Kapot Tracker

Бортовий журнал вашого авто: витрати, заправки, розхід пального та сервісні інтервали — все в одному застосунку.

## Архітектура

- **Backend** — FastAPI (Python 3.12), REST API з JWT-авторизацією. База даних: SQLite за замовчуванням для локальної розробки, PostgreSQL у Docker/продакшені (SQLAlchemy, `postgresql+psycopg2`).
- **Frontend** — React (Vite) як PWA. У Docker збирається у статичні файли та роздається через nginx, який проксіює `/api` на backend.
- **Авторизація** — JWT (access-токени, термін дії налаштовується через `ACCESS_TOKEN_EXPIRE_MINUTES`).

```
frontend (React PWA, nginx :80) ──/api──> backend (FastAPI :8000) ──> PostgreSQL / SQLite
```

## Можливості MVP

- Авторизація: реєстрація та вхід (JWT).
- Гараж: кілька авто на одного користувача.
- Журнал записів 4 типів: заправка, ТО, ремонт, витрата.
- Розхід пального за методом «повний-до-повного».
- Сервісні інтервали з прогнозом дати наступного обслуговування.
- Аналітика з графіками: витрати за категоріями та періодами, динаміка розходу.
- PWA: встановлення на телефон/десктоп, робота як окремий застосунок.
- OCR чеків: фото чека з заправки автозаповнює форму запису.
- Telegram-бот: пробіг, швидкі витрати та нагадування про ТО прямо з месенджера.
- PDF-звіт сервісної історії авто — зручно як передпродажна історія авто.
- Прогнозна аналітика: середні витрати на місяць, прогноз поточного місяця, найближчі ТО з орієнтовною вартістю.
- Детальна сторінка запису: перегляд усіх полів, редагування, дублювання («Повторити»), видалення.
- Фото до записів (чеки, акти виконаних робіт): галерея з лайтбоксом, до 10 МБ на файл.
- Пошук по журналу: нотатки, роботи ТО, категорія/деталь ремонту, АЗС — поєднується з фільтром типу.
- Редагування сервісних інтервалів та швидке оновлення пробігу прямо з дашборда.
- Alembic-міграції схеми БД — застосовуються автоматично на старті.
- Щоденний бекап бази даних з ротацією та доставкою у Telegram.
- Експорт та імпорт даних: повний JSON-дамп акаунта (портативний), CSV журналу авто.
- Відновлення пароля через привʼязаний Telegram (одноразовий 6-значний код).
- Захист від перебору: rate limiting на вході, реєстрації та скиданні пароля.
- Кнопка «Виконано» на інтервалі: один клік записує ТО і зсуває відлік уперед.
- Пресети інтервалів: типові інтервали ТО та українські документи (ОСЦПВ, техогляд, зелена карта, податок).
- Заправка за 20 секунд: чипси останніх АЗС, підказка останнього пробігу, попередження про дивні значення, розхід одразу в стрічці.
- Категорії витрат (мийка, паркування, штраф, страхування, податок, шини, аксесуари); сума за кожною категорією видно в аналітиці, у картці «За категоріями».
- Офлайн-режим: запис на АЗС без звʼязку лягає в чергу IndexedDB і сам відправляється, щойно мережа повернулась.
- Діагностика OBD: імпорт CSV-логів Car Scanner з вердиктами по сажовому фільтру, форсунках і АКБ.
- Тех. довідка: моменти затяжки, обʼєми рідин і допуски під капотом — з пресетом для Golf 7 1.6 TDI.
- Бібліотека документів: техпаспорт, страховка, техогляд, рахунки (PDF та фото); страховка з датою закінчення сама створює нагадування.
- VIN і держномер авто з офлайн-декодером VIN (виробник, країна, модельний рік).
- «Мої АЗС»: скільки залито, витрачено і яка витрата на кожній заправці.

## Діагностика OBD

Розділ «Діагностика» перетворює лог з [Car Scanner ELM OBD2](https://www.carscanner.info/) на висновки про здоровʼя авто. Як експортувати лог: **Налаштування → Записи → Експорт CSV**, потім перетягнути файл у дроп-зону на `/diagnostics`.

`POST /api/cars/{car_id}/obd` (multipart CSV, ≤20 МБ) парсить файл і зберігає сесію. Парсер (`app/services/obd.py`) стійкий до реальних експортів:

- роздільник `,` або `;` визначається автоматично, десяткова кома нормалізується;
- час — абсолютний ISO або секунди від старту;
- назви колонок зіставляються з канонічними метриками за нечітким збігом синонімів (укр/англ), тому профілі різних версій і мов Car Scanner читаються однаково;
- сміттєві значення відкидаються санітарними межами (сажа 0..100 г, напруга 6..18 В, температура −40..150 °C, корекція форсунки −10..10 mm³).

Вердикти (українською, у відповіді сесії):

| Вердикт | 🟡 Увага | 🔴 Критично |
|---|---|---|
| Сажовий фільтр (DPF) | сажа > 18 г — регенерація скоро | сажа > 24 г |
| Форсунки | розкид корекцій > 3 mm³ | будь-яка корекція > 5 mm³ |
| АКБ | < 12.2 В у спокої — недозаряд | < 9.6 В при старті — просідає |

**Обмеження, про які варто знати:**

- **Сирий CSV не зберігається.** У базу лягає тільки зведення по метриці (min/max/avg/last) і графік, проріджений до ≤200 точок зі збереженням екстремумів. Оригінальний файл після імпорту не відновити — зберігайте його самі, якщо він потрібен.
- **Розпізнаються лише відомі колонки.** Метрики поза списком синонімів потрапляють у `unmapped_columns` (інтерфейс покаже «не розпізнано: N колонок») і не зберігаються як метрики. Дані не губляться мовчки, але й не аналізуються.
- Вердикти — це індикатор «глянути уважніше», а не діагноз замість сервісу.

## Тех. довідка

Картка «Тех. довідка» на авто в «Гаражі» веде на сторінку `/garage/{carId}/specs`: моменти затяжки, рідини та обʼєми, допуски, інше — згруповано за категоріями, з інлайн-редагуванням.

Кнопка «Підставити типові для Golf 7 1.6 TDI» (`POST /api/cars/{car_id}/specs/preset?key=golf7_16tdi`) створює 9 стартових рядків і показується лише поки специфікацій немає. Пресет ідемпотентний за парою (категорія, назва) — повторний виклик не дублює рядки.

**Обмеження:** глобальної бази специфікацій немає і не планується — підтримувати її для всіх моделей нереально. Пресет існує один (Golf 7 1.6 TDI, з реального сервісного паспорта), решта авто заповнюється вручну. Усі значення, включно з пресетними, редаговані.

## Документи

Бібліотека документів на авто: техпаспорт (`tech_passport`), страховка (`insurance`), техогляд (`inspection`), рахунок (`invoice`), інше (`other`). Приймаються зображення та PDF, до 10 МБ; файли лежать у тому ж сховищі, що й фото записів (`UPLOADS_DIR`).

**Звʼязок з нагадуваннями:** якщо у страховки або техогляду заданий термін дії (`expires_at`), у тій самій транзакції створюється річний сервісний інтервал «{назва} (документ)» — нагадування прийде за 14 днів до кінця. Id інтервалу повертається у відповіді (`linked_interval_id`). Видалення документа інтервал не чіпає: строк не зникає разом із файлом.

## VIN і держномер

У формі авто є поля «VIN» і «Держномер». VIN зберігається у верхньому регістрі, валідуються рівно 17 символів з набору `A-HJ-NPR-Z0-9` (без I, O, Q). **Контрольна цифра не перевіряється** — європейські VIN масово її не мають (саме випадок Golf 7 `WVWZZZAUZHP541983`). Номер зберігається як введений, без формат-валідації: бувають і європейські, і транзитні.

`POST /api/vin/decode` — чистий офлайн-декодер (`app/services/vin.py`, без мережі): WMI-таблиця дає виробника і країну, 10-й символ — модельний рік за ISO 3779. Після введення 17 символів форма показує підказку «Volkswagen, Німеччина, 2017 — підставити рік?»: рік підставляється лише кнопкою, мовчки нічого не перезаписується.

**Обмеження:** WMI-таблиця покриває поширені в Україні марки, а не весь світ; невідомий WMI повертає `valid: true` з `manufacturer: null`.

## Мої АЗС

Список «Мої АЗС» під графіками аналітики (зʼявляється від 2 заправок) показує по кожній станції: кількість заправок, залиті літри, витрачені гроші, середню ціну за літр і середню витрату.

Дві деталі, які легко зрозуміти неправильно:

- **Витрата рахується по відрізках, що починаються на цій АЗС** — пальне, спалене за відрізок, було куплене саме на його початку. Станція, яка ніколи не відкривала виміряного відрізка (лише часткові заправки або остання заправка в історії), показує «—».
- **Назви групуються без урахування регістру** («okko» і «OKKO» — одна станція), канонічною стає найчастіше вживана; заправки без назви збираються в групу «Без назви».

## OCR чеків

Як це працює: ви фотографуєте чек — фронтенд надсилає фото на `POST /api/ocr/scan` — розпізнаний текст (сума, літри, дата) автозаповнює форму нового запису, залишається лише перевірити й зберегти.

Розпізнавання виконує [Tesseract](https://github.com/tesseract-ocr/tesseract) (українська + англійська мови):

- **Локальна розробка**: потрібен системний бінарник tesseract. На macOS:

  ```bash
  brew install tesseract tesseract-lang
  ```

- **Docker**: нічого робити не треба — tesseract уже встановлено в образ backend.
- Якщо бінарник не знайдено, ендпоінт відповідає `503`, решта API працює як звичайно.

## Telegram-бот

Бот дозволяє вести журнал, не відкриваючи застосунок.

Налаштування:

1. Створіть бота через [@BotFather](https://t.me/BotFather) і отримайте токен.
2. Впишіть `TELEGRAM_BOT_TOKEN` і `TELEGRAM_BOT_USERNAME` (без `@`) у `.env`.
3. Запустіть бота:

   ```bash
   # Локально (у venv backend-а)
   python -m app.bot.main

   # Docker (окремий сервіс, не стартує при звичайному up)
   docker compose --profile bot up
   ```

Можливості:

- Привʼязка акаунта через код із розділу «Гараж» (діє `LINK_CODE_EXPIRE_MINUTES` хвилин).
- `/status` — стан кожного авто: пробіг і до трьох найближчих сервісних інтервалів.
- `/report` — PDF-звіт сервісної історії документом у чат (за кількох авто бот спитає, яке).
- Оновлення пробігу: просто надішліть число.
- Швидкі витрати одним повідомленням: `мийка 300`.
- Заправка одним повідомленням: `заправка 45л 2500` — літри плюс сума або ціна за літр, третє значення бот рахує сам (як і OCR чеків).
- Фото чека прямо в чат: бот розпізнає заправку (той самий tesseract, що й `/api/ocr/scan`), показує прочитане і зберігає фото до запису після підтвердження.
- Нагадування про наближення сервісних інтервалів (ТО) з кнопками «Виконано» (записує ТО і починає відлік заново) та «Нагадати через 7 днів».

Кожен запис бот створює лише після підтвердження кнопкою — спочатку показує, що саме зрозумів, і за кількох авто питає, кому це записати. Пробіг рухається тільки вперед: менше за поточне значення бот не приймає.

### Бот і спільний доступ

- У боті видно **всі доступні авто** — свої та спільні; у списку вибору спільні позначені «(спільне)», щоб заправка не потрапила не тому авто.
- Кожен запис, створений через бота, зберігає **автора** — у журналі видно, хто саме його додав.
- **Редактор** може все те саме, що й власник: пробіг, витрати, заправки, фото чеків. **Спостерігач** отримує ввічливу відмову і підказку попросити власника змінити роль; читання (`/status`, `/report`) йому лишається.
- **Нагадування про ТО отримує лише власник авто — це свідоме рішення, а не недогляд.** Інтервал — це один факт про одне авто, а учасників у спільного авто кілька: розсилка всім перетворила б одну заміну оливи на повідомлення кожному члену сімʼї, і то на кожному колі нагадувань. До того ж кнопки «Виконано» / «Нагадати через 7 днів» діють на спільний інтервал — перший, хто натисне, мовчки відповість за все повідомлення, які інші ще читають. Одне авто — один власник — одне нагадування. Учасники нічого не втрачають: той самий стан вони бачать у `/status` будь-коли, і про **власні** авто нагадування їм приходять як завжди. Якщо колись знадобиться інакше — правильна форма це персональна підписка учасника, а не просто ширший запит.

## Сервісні інтервали: «Виконано» та пресети

Кнопка **«Виконано»** на картці інтервалу (дашборд) закриває коло обслуговування одним рухом: `POST /api/intervals/{interval_id}/complete` в одній транзакції створює запис ТО і зсуває інтервал уперед (`last_odometer`, `last_date`), а відповідь повертає і створений запис, і оновлений інтервал — `{"log": ..., "interval": ...}`. Форма компактна: пробіг авто і сьогоднішня дата вже підставлені, вартість = запчастини + робота, а роботи за замовчуванням дорівнюють назві інтервалу — щоб запис говорив, що саме зробили. Та сама дія доступна кнопкою «Виконано» під нагадуванням у Telegram.

**Пресети** (`GET /api/interval-presets`) — дві готові групи для авто, у якого інтервалів ще немає (кнопки в «Гаражі»):

- `maintenance` — типові інтервали ТО: олива двигуна, повітряний/паливний/салонний фільтр, ГРМ, гальмівна рідина (за пробігом, деякі з календарним запобіжником).
- `compliance` — українські документи: поліс ОСЦПВ, техогляд, зелена карта, транспортний податок (тільки за датою — поліс спливає в дату, а не на пробігу).

## Щоденна заправка

Заправка — найчастіша дія в застосунку, тож форма готується заздалегідь. `GET /api/cars/{car_id}/refuel-context` одним запитом віддає підказки: `recent_stations` (до пʼяти останніх АЗС — чипси під полем), `last_price_per_liter` (підставляється в ціну), `last_refuel_odometer`, `last_entry_odometer` і `last_entry_date` (підказка «Останній запис: 123 456 км · 10.07.2026»).

**Попередження, а не блокування.** Дивні значення форма підсвічує, але зберегти дозволяє завжди — і історичний запис, і передоплачений чек є нормою:

- пробіг менший за останній запис — «це історичний запис?»;
- пробіг не змінився з останнього запису — «розхід не порахується»;
- дата в майбутньому.

**Розхід одразу в стрічці.** `refuel.consumption_l_100km` — це сегмент «повний-до-повного», що *закінчується* цією заправкою: null для неповних баків, для першої заправки в сегменті та коли відстань не вимірюється. Значення рахується за всю історію авто одним проходом (сегмент може виходити за межі сторінки чи фільтра), тому воно однакове в списку, в деталях запису і в історії аналітики.

## Офлайн-режим

На АЗС звʼязку часто немає, а запис має пережити дорогу додому. Тому нові записи журналу пишуться в чергу IndexedDB (`src/offline/outbox.js`, без зайвих залежностей) і відтворюються FIFO, щойно мережа повернулась; у стрічці вони позначені як «Очікує синхронізації». Смуга вгорі показує стан звʼязку — бурштинова офлайн, коротка зелена після повернення.

Конфлікти вирішує сервер: черга не відкочує пробіг авто назад (бекенд і так рухає його тільки вперед), а після відправки клієнт просто перечитує дані. Помилка «спробуйте пізніше» (401, 408, 429) лишає запис у черзі — сесія, що протухла, не має коштувати людині набраного запису; тільки справжня відмова валідації прибирає його з черги.

PWA встановлюється як застосунок: `manifest.webmanifest` з PNG-іконками 192/512 і maskable-512, `apple-touch-icon` для iOS, офлайн-оболонка через service worker (`vite-plugin-pwa`, Workbox).

## PDF-звіт та прогноз

`GET /api/cars/{car_id}/report` формує PDF-звіт сервісної історії авто (reportlab, вбудовані шрифти DejaVu — кирилиця відображається коректно): дані авто, поточний пробіг, повна історія записів (заправки, ТО, ремонти, витрати) та підсумки витрат. Зручно показати покупцеві як передпродажну історію авто. В інтерфейсі кнопка «Звіт PDF» є у шапці розділу «Аналітика» та на картці авто в «Гаражі».

Прогнозна аналітика (ключ `forecast` у відповіді `/api/cars/{car_id}/analytics`) відображається в секції «Прогноз» розділу «Аналітика»: середні витрати на місяць, прогноз поточного місяця та найближчі ТО з орієнтовною вартістю.

## Міграції (Alembic)

Схема бази даних версіонується Alembic (каталог `backend/alembic/`). Запускати міграції вручну не потрібно: і API (lifespan у `app/main.py`), і бот викликають `app.migrations.run_migrations` на старті, яка:

1. Штампує стару базу (створену ще `create_all`, без таблиці `alembic_version`) базовою ревізією `0001` — таблиці не перестворюються.
2. Виконує `alembic upgrade head` — свіжа база отримує всю схему, наявна — лише нові ревізії.
3. Викликає `ensure_schema` як фолбек для старих dev-баз без окремих колонок.

У Docker це працює так само — контейнер backend мігрує базу під час старту, окремий крок деплою не потрібен.

Робота з міграціями вручну (з каталогу `backend`, у venv):

```bash
alembic upgrade head                                  # застосувати всі ревізії
alembic revision --autogenerate -m "опис змін"        # нова міграція зі змін у models.py
alembic history                                       # список ревізій
```

URL бази Alembic бере з `DATABASE_URL` (той самий `.env`, що й застосунок).

## Бекапи

Модуль `app/backup.py` створює бекап бази у `BACKUP_DIR`:

- **SQLite** — гарячий бекап через `sqlite3.Connection.backup()` (безпечно при запущеному uvicorn), файл `kapot_tracker-YYYYMMDD-HHMMSS.db`.
- **PostgreSQL** — `pg_dump`, файл `kapot_tracker-YYYYMMDD-HHMMSS.sql.gz` (у Docker-образі `pg_dump` уже встановлено).

Ротація лишає `BACKUP_KEEP` (за замовчуванням 14) найновіших файлів. Якщо заданий `BACKUP_TELEGRAM_CHAT_ID`, бекап додатково надсилається документом у цей Telegram-чат — **тільки адмінський чат**: бекап містить дані всіх користувачів сервісу.

Запуск:

```bash
# Вручну або з cron (у venv backend-а)
python -m app.backup

# Docker
docker compose exec backend python -m app.backup
```

Бот-процес (`--profile bot`) додатково робить бекап автоматично раз на добу після розсилки нагадувань. У Docker бекапи зберігаються в named volume `backups`; основна офсайт-копія — доставка у Telegram.

Відновлення: для SQLite — зупинити застосунок і замінити файл бази на файл бекапу; для PostgreSQL — `gunzip -c kapot_tracker-....sql.gz | psql <DSN>` у порожню базу.

## Експорт та імпорт даних

На відміну від бекапів (адмінська копія всієї бази), експорт — це особиста копія даних користувача:

- **`GET /api/export`** — повний JSON-експорт акаунта (файл `kapot-tracker-export-YYYYMMDD.json`): усі авто з сервісними інтервалами та журналом. Дамп не містить внутрішніх id, тому він портативний — його можна імпортувати в інший акаунт чи інший інстанс. Фото подорожують лише як метадані (назва файлу, тип, розмір) — самі файли в експорт не пакуються.
- **`GET /api/cars/{car_id}/export.csv`** — журнал одного авто в CSV (utf-8-sig, відкривається в Excel без проблем із кирилицею); роботи ТО в колонці `items` розділені `«; »`.
- **`POST /api/import`** — імпорт JSON-експорту в поточний акаунт. Політика v1 — append: усе створюється як нові записи, наявні дані не змінюються (зокрема пробіг існуючих авто). `schema_version != 1` або перший невалідний елемент → 422 із зазначенням шляху (`cars[0].logs[3]: ...`), при цьому не створюється нічого — імпорт виконується в одній транзакції.

В інтерфейсі: розділ «Гараж», картка «Дані» — кнопки «Експортувати все (JSON)» та «Імпортувати з файлу» (перед імпортом показується підтвердження: скільки авто, записів та інтервалів буде додано); кнопка «CSV журналу» — на картці авто.

## Відновлення пароля

Скидання пароля працює через Telegram — **потрібен привʼязаний Telegram-акаунт** (привʼязка робиться заздалегідь у розділі «Гараж»). Пошта не використовується.

1. На сторінці входу натисніть «Забули пароль?» (`/reset`) і введіть email.
2. `POST /api/auth/reset/request` завжди відповідає 202 — з відповіді неможливо дізнатися, чи існує акаунт. Якщо акаунт існує і привʼязаний Telegram, бот надсилає 6-значний код: діє 10 хвилин, одноразовий, у базі зберігається лише його bcrypt-хеш.
3. Введіть код і новий пароль (мінімум 8 символів) — `POST /api/auth/reset/confirm`. Будь-яка причина відмови (невірний код, прострочений, вже використаний) дає однакову відповідь 400 «Невірний або прострочений код».

Раніше видані JWT-токени лишаються чинними до власного терміну дії — скидання пароля не відкликає активні сесії.

## Захист від перебору

Чутливі auth-ендпоінти захищені in-memory sliding-window rate limiter-ом (`app/ratelimit.py`). Перевищення ліміту → `429 {"detail": "Забагато спроб. Спробуйте пізніше."}` із заголовком `Retry-After`:

| Ендпоінт | Ліміт | Ключ |
|---|---|---|
| `POST /api/auth/token` | 5 спроб / 5 хв; успішний вхід скидає лічильник | IP + email |
| `POST /api/auth/register` | 3 / год | IP |
| `POST /api/auth/reset/request` | 3 / 15 хв | IP + email |
| `POST /api/auth/reset/confirm` | 5 / 15 хв | IP + email |

IP клієнта береться з першого елемента `X-Forwarded-For` (у Docker nginx проставляє його: `proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;` у `frontend/nginx.conf`), без проксі — адреса сокета. Лічильники живуть у памʼяті процесу, що відповідає одно-воркерному деплою.

## Огляд API

| Група ендпоінтів | Призначення |
|---|---|
| `/api/auth` | Реєстрація (`/register`), вхід (`/token`), профіль (`/me`), скидання пароля через Telegram (`/reset/request`, `/reset/confirm`) |
| `/api/cars` | CRUD авто в гаражі; поля `vin`, `plate`, `avg_daily_km` (ефективний), `avg_daily_km_computed`, `avg_daily_km_override` |
| `/api/vin/decode` | Офлайн-декодер VIN: виробник, країна, модельний рік (без мережі, контрольна цифра не перевіряється) |
| `/api/cars/{car_id}/logs`, `/api/logs/{log_id}` | Журнал записів (заправка, ТО, ремонт, витрата); параметр `q` — пошук по нотатках, роботах ТО, ремонту та АЗС; у заправці `refuel.consumption_l_100km` — розхід сегмента, у витраті `expense.category` — категорія |
| `/api/cars/{car_id}/refuel-context` | Підказки для форми запису: останні АЗС, остання ціна за літр, останній пробіг і дата |
| `/api/logs/{log_id}/photos`, `/api/photos/{photo_id}` | Фото записів: завантаження (multipart, `image/*`, ≤10 МБ), отримання файлу, видалення |
| `/api/cars/{car_id}/intervals`, `/api/intervals/{interval_id}` | Сервісні інтервали та прогноз наступного ТО |
| `/api/intervals/{interval_id}/complete` | «Виконано»: запис ТО + зсув інтервалу в одній транзакції → `{"log", "interval"}` |
| `/api/interval-presets` | Готові набори інтервалів: `maintenance` (типові ТО) і `compliance` (ОСЦПВ, техогляд, зелена карта, податок) |
| `/api/cars/{car_id}/analytics` | Агреговані дані для графіків, у т.ч. розхід пального (повний-до-повного); ключ `forecast` — прогнозна аналітика, `expense_by_category` — витрати за категоріями, `stations` — статистика по АЗС (витрата рахується по відрізках, що починаються на станції) |
| `/api/cars/{car_id}/specs`, `/api/specs/{spec_id}` | Тех. довідка авто: моменти затяжки, рідини, допуски (CRUD) |
| `/api/cars/{car_id}/specs/preset?key=golf7_16tdi` | Пресет тех. довідки — 9 стартових рядків, ідемпотентний за (категорія, назва) |
| `/api/cars/{car_id}/documents`, `/api/documents/{document_id}` | Документи авто: завантаження (multipart, `image/*` та `application/pdf`, ≤10 МБ), список, стрім файлу, видалення; страховка/техогляд з `expires_at` створює інтервал → `linked_interval_id` |
| `/api/cars/{car_id}/obd`, `/api/obd/{session_id}` | Імпорт CSV Car Scanner (multipart, ≤20 МБ), список сесій, сесія з метриками та вердиктами, видалення |
| `/api/cars/{car_id}/report` | PDF-звіт сервісної історії авто (reportlab, вбудовані шрифти DejaVu) |
| `/api/ocr/scan` | Розпізнавання фото чека (tesseract) для автозаповнення форми запису |
| `/api/telegram/*` | Привʼязка Telegram-акаунта: генерація коду, статус, відвʼязка |
| `/api/export`, `/api/import` | Повний JSON-експорт акаунта та append-імпорт (портативний формат без внутрішніх id) |
| `/api/cars/{car_id}/export.csv` | CSV-експорт журналу авто (utf-8-sig, Excel-сумісний) |
